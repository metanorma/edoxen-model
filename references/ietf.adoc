IETF Datatracker Meeting Data
=============================
IETF (Internet Engineering Task Force) holds three numbered
meetings per year ("IETF NNN"). Each meeting is a multi-day,
multi-track gathering of ~1,000 attendees in parallel rooms,
plus Birds of a Feather (BOF) sessions, tutorials, plenaries,
hackathons, and side meetings.

The Datatracker (datatracker.ietf.org) is the canonical
machine-readable record: each session has a JSON record,
a structured Markdown agenda, minutes, slide decks, and
presentations. It is the best-structured public meeting
data available — and the most diverse (working groups,
research groups, teams, directorate, plenary, BOFs).

Edoxen adopts IETF meeting data as the canonical multi-day,
multi-track, multi-room, multi-session meeting reference.
Where standards-body references (ISO, ITU) are *single-track
plenary meetings with a published resolution record*, IETF
meetings are *multi-track weeks with N parallel sessions, each
its own meeting-within-a-meeting*.

|===

  - SD-JWT VC - Brian (20 min)

Note the structure:

- Agenda grouped by **day** (Tuesday / Friday)
- Each day has a **duration budget** (120 min)
- Each agenda item has **presenter** + **duration** + optional
  **linked document** (draft URL)
- Each session has an **agenda URL** AND a **minutes URL** AND
  slide decks per presentation

This is the real shape of an IETF agenda — multi-day, with
duration budgets, presenter attributions, and structured per-item
links.


== What IETF data forces onto the Edoxen Meeting model

Concrete additions:

- `Meeting.session_kind: Enum {wg, rg, team, ietf, rag, irtf,
  directorate, bof}` — discriminator for what KIND of meeting
- `Meeting.area: String?` — parent area (sec/rtg/ops/...) for
  hierarchical grouping
- `Meeting.committee_state: Enum {active, bof, concluded}` —
  lifecycle of the group, not just the meeting
- `Meeting.is_bof: Boolean` — explicit Birds-of-a-Feather flag
  (BOFs propose new working groups; need to be tracked)
- `ScheduleItem.start_time: DateTime` (today Edoxen has
  `ScheduleItem.date` + `ScheduleItem.time` as a String; this
  is ambiguous)
- `ScheduleItem.duration: Duration` — explicit duration; today
  implicit
- `ScheduleItem.presenter: Person` — agenda-item presenter
  attribution (today: anonymous "event" string)
- `ScheduleItem.linked_documents: URL[]` — drafts/papers/inputs
- `AgendaItem.presenter: Person` (already partially in Edoxen
  via `AgendaItem.title`, but should be a structured Person)
- `AgendaItem.duration: Duration`
- `AgendaItem.linked_documents: URL[]`
- `AgendaItem.attachments: Attachment[] {name, rev, url, type}`
  — for slide decks, drafts, jabber logs

Crucial structural difference from standards-body references:

**IETF meetings have no "Resolution" layer.** IETF does NOT
produce resolutions — its output is RFCs (Internet-Drafts that
go through a separate standards process). IETF meetings produce:

- Minutes (running narrative, present at `minutes` URL)
- Consensus calls (record in minutes)
- Document updates (drafts that move between states)
- Decisions on next steps (record in minutes)

The Edoxen `ResolutionCollection` join (via `resolution_refs[]`)
is therefore *absent* in IETF data — but the `Minutes` is
present and richly populated. This is a clean test of whether
Edoxen's ResolutionResolution side is genuinely orthogonal
(pluggable) or secretly required.

Conclusion from the data: the `ResolutionCollection` link is
optional for meetings that don't produce resolutions. Edoxen's
Meeting should not REQUIRE `resolution_refs`.


== What IETF does NOT have (and Edoxen must keep)

1. **Resolutions / formal decisions.** IETF runs on consensus
   in the room, recorded in minutes, with formal RFC output
   through a separate process. No "Resolution" first-class
   entity.

2. **Motions, divisions, voting.** IETF uses "hums" and
   polling; not formal roll-call votes. The closest is
   consensus calls in minutes.

3. **Fixed committee structure.** Groups come and go (BOFs,
   concluded WGs). The `Meeting.committee_state` field
   captures this; standards-body Edoxen has assumed long-
   lived committees.

4. **Permanent officers.** Chairs rotate (every 2 years);
   no permanent "secretary" in the IETF context. Area
   Directors are appointed by the Nominating Committee.


== Multi-track structure

The IETF model has THREE levels:

1. **Meeting** (IETF 122, the whole week)
2. **Session** (OAuth WG session #1, OAuth WG session #2)
3. **AgendaItem** (Token Status List, Attestation, etc.)

Edoxen's current model conflates (1) and (2) — every Meeting
is itself a trackless event. IETF data shows that real meetings
have multiple parallel sessions in the same week, sometimes
the same group meeting twice (Tuesday OAuth, Friday OAuth).

The right model:

  MeetingSeries {
    name: "IETF 122"           // = the whole week
    start_date, end_date
    venue: Location
    sessions: Meeting[]        // = each WG/RG/team/BOF session
  }

  Meeting {                    // = one session within a series
    parent_series: MeetingSeries
    track: String?             // = the WG acronym, RG acronym
    area: String?              // = sec/rtg/ops parent
    type: MeetingType          // = wg/rg/team/ietf/bof
    agenda: Agenda
    minutes: Minutes
    presentations: Presentation[]
  }

Or alternatively, keep `Meeting` as the series and add a new
`Session` entity:

  Meeting (the whole IETF 122)
    sessions: Session[]        // each session within the week

The IETF data structures sessions explicitly, so adding a
`Session` (or `Track`) entity to Edoxen is the natural move.


== Status enum (from `agenda.json`)

Field `status` on each session. Observed values:

- `sched`     — scheduled (the default)
- `resched`   — rescheduled (time/room changed)
- `canceled`  — cancelled
- (missing)   — not yet scheduled

Maps onto Edoxen's current `MeetingStatus` enum
(`planned` / `confirmed` / `cancelled` / `completed` /
`in_session`). Edoxen today has no `rescheduled` state —
IETF data forces adding it.

Also from Datatracker: sessions that DID happen get a
`minutes` URL; sessions that didn't (cancelled) don't. This
is the simplest possible "did this meeting produce a record"
flag.

multi-track pattern that Edoxen doesn't currently model.


== Net contribution to Edoxen

IETF is the *multi-day, multi-track, multi-session meeting
with linked documents and structured presentations* reference.
Adopting it forces:

- A `MeetingSeries` (or `Session`) entity to handle parallel
  tracks within one overall event
- A `Meeting.session_kind` enum
- A `Meeting.area` field
- A `Meeting.committee_state` lifecycle enum
- A `Meeting.is_bof` flag
- `ScheduleItem.start_time: DateTime` + `duration: Duration`
  (replace ambiguous `time: String`)
- `ScheduleItem.presenter: Person`
- `ScheduleItem.linked_documents: URL[]`
- `AgendaItem.presenter: Person`
- `AgendaItem.duration: Duration`
- `AgendaItem.linked_documents: URL[]`
- `Meeting.status` enum extended with `rescheduled`
- `Meeting.resolution_refs[]` made OPTIONAL (not required),
  to accommodate non-resolution-producing meetings

The most consequential single addition is `MeetingSeries` /
`Session`. Every multi-day conference (W3C TPAC, ApacheCon,
RSA Conference, etc.) requires it. Standards-body plenary
meetings (ISO/TC 154 Nth plenary) can stay as a single
`Meeting` since they don't have parallel sessions — backward-
compatible.
