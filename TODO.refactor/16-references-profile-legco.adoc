= 16 — references/profiles/legco.adoc — first profile

== Why

Proves the profile mechanism works end-to-end. The HK LegCo
profile (from `references/hansard.adoc`) is the canonical
example — 8 extension kinds mapping LegCo's OData schema
onto Edoxen core.

== File to create

- `references/profiles/legco.adoc`

== Profile namespace

`profile = "legco"`

== Extension kinds

| kind | Hangs on | Maps to LegCo entity |
|---|---|---|
| `bill` | Topic.documents[] (as TopicDocument with kind="bill") | Bills + BillStages |
| `petition` | Topic.documents[] | Petitions |
| `address` | Meeting.components[] (kind="address") | Addresses |
| `statement` | Meeting.components[] (kind="statement") | Statements |
| `question` | Topic.documents[] OR Meeting.extensions[] | Questions (oral/written/urgent) |
| `voting-result` | Meeting.votings[] (via extension) | VotingResults |
| `summoning-bell` | Meeting.extensions[] | SummoningBells |
| `subject` | Topic (resumption_of) | Subjects (with ResumptionOf) |

== Sample profile extension (YAML)

```yaml
extensions:
  - profile: legco
    kind: bill
    ref: urn:legco:bill:2024-waste-disposal-amendment
    attributes:
      - key: bill_number
        value: "2024年第87号"
      - key: stage
        value: second_reading
      - key: introduced_by
        value: Government
```

== Documentation structure

- Profile namespace declaration
- Each kind with: definition, where it hangs, attributes, sample YAML
- Mapping table to source LegCo OData entities
